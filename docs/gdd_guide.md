# GDD Guide (внутренний гайд)

Короткая, практичная инструкция по написанию Game Design Document (GDD) для наших игр.

## 1. Что такое GDD и зачем

GDD — живой документ, который описывает замысел, правила и требования к игре
достаточно точно, чтобы команда могла реализовать и поддерживать ее без догадок.
Документ обновляется по мере изменения дизайна и масштаба.

Зачем он нужен:
- фиксирует видение и ограничения;
- снижает риск расползания фич;
- дает общий язык для дизайнеров, программистов и артистов;
- ускоряет тестирование, т.к. критерии ожидаемого поведения ясны заранее.

## 2. Принципы хорошего GDD

- Достаточная детализация: без воды, но с конкретикой по правилам и состоянию.
- Однозначность: любое спорное правило должно быть описано так, чтобы его можно было
  реализовать однозначно.
- Живой формат: версия + дата изменений + список ключевых решений.
- Иерархия: от высокого уровня (питч) до конкретных правил и таблиц.
- Проверяемость: каждое правило должно быть верифицируемым тестом или сценарием.

## 3. Рекомендуемая структура (полная)

1) Executive Summary
   - название, короткий питч, дизайн-пиллары.
2) Цели и аудитория
   - кто играет и зачем.
3) Core Loop
   - основной игровой цикл в 3–6 шагов.
4) Механики и правила
   - движение, столкновения, ресурсы, прогрессия, поражения.
5) Контент и генерация
   - тайлы, враги, предметы, вероятности, ограничения, правила генерации.
6) Уровни/сцены/структура мира
   - как строится “бесконечный” или конечный прогресс.
7) UI/UX и управление
   - расположение контролов, HUD, обучение.
8) Визуальный стиль
   - палитра, стиль арта, требования к читабельности.
9) Аудио
   - музыка/саунд-эффекты, когда и зачем играют.
10) Технические требования
   - производительность, платформы, API, сохранения.
11) Аналитика и метрики
   - что измеряем (score, ретеншн, сессия).
12) План реализации
   - инкрементные шаги, каждая версия играбельна.
13) Риски и допущения
   - что может сломать баланс/план.

## 4. Минимальный GDD (для старта)

Если нужно начать быстро, достаточно 1–2 страниц:
- Короткий питч + дизайн-пиллары
- Core Loop
- Список механик с краткими правилами
- Условия победы/поражения
- UI/контролы (минимум)
- План из 3–5 шагов (каждый шаг играбелен)

## 5. Обязательные разделы для наших игр

Для репозитория eoe всегда включаем:
- “Core Loop”
- “Механики и правила” с явными условиями поражения
- “UI/UX и управление” (портрет, нижняя треть)
- “Генерация/контент” (если бесконечный режим)
- “Технические требования” (Phaser, Game API, производительность)
- “План реализации” с инкрементами

## 6. Глубина детализации (сколько писать)

Правило: любая механика должна быть описана через:
- триггер (когда срабатывает),
- эффект (что меняется в state),
- визуальную реакцию (что видит игрок).

Если правило не тестируется или не влияет на state — его в GDD быть не должно
или оно должно стать “пожеланием/идеей”.

## 7. Формат и обновления

Рекомендуемый формат:
- заголовки H1–H3;
- таблицы только для вероятностей/баланса;
- раздел “Changelog” (дата, кто изменил, что изменил).

При обновлениях:
- не удаляйте старые решения без фиксации причины;
- фиксируйте изменения, влияющие на API или state.

## 8. Чек-лист качества GDD

- Питч понятен за 30 секунд?
- Есть ли ясный Core Loop?
- Можно ли проверить каждую механику тестом?
- Явно описаны условия поражения?
- UI понятен без прототипа?
- Есть ли план инкрементов?
- Нет ли скрытых “магических” правил?

## 9. Типовые ошибки

- Перепутаны концепт и реализация (слишком ранние детали движка).
- Нет определения проигрыша/победы.
- Механики описаны словами, но без формализованных условий.
- Слишком большой документ без приоритета.
